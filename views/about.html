About the Site

This is a simple site meant to show off the functionality of the Vanilla SPA framework ©. I encourage you to take a look and change it to fit your needs.


About the Framework

A Single Page Application is just a single web page that loads other pages into it. You can achieve some cool effects and speed up production ( After initial development ) using this method. There are alot of sites that utilize this method who I will not mention here.

This framework includes some pre-built modules you can use to get started. They are simple scripts to illustrate how you should build your views and maybe use some if they are useful to you.


The framework uses vanilla JavaScript( ES2022 ), standard CSS, and HTML.


Vanilla SPA framework © uses a global object to store data that is wiped every page load. You can store anything you can store inside an object - Functions, Other Objects, variables etc. Below is a quick and dirty rundown on how it works.


Folder Structure

The folder structure is as follows:

root
├ main.html
├ /views
├ /css
├ /img
└ /js
   ├ /framework
   ├ /modules
   └ app.js


main.html

The main page of the site. All content will be loaded into the "#content" element.


/views

This is where you would store the .html files with .css / module tags set like normal.


/css

This is where you would store the .css files for your views.


/img

This is where you would store the image files for your views.


/js

The main directory for your scripts.


/framework

This is where the framework modules live, do not mess with these unless you are customizing.


/modules

This is where you would store all your individual modules for views.


app.js

The main entry point of the app. You can use this as a psudo-config file as there are some properties you can set here as needed. ( More info inside the actual file, or the docs )


HTML Views (.html files)

These are your standard HTML files you would see anywhere. You would add your style tags and module tags normally, the framework will extract these and inject them into the main page, and unload them when appropriate.

* The body is what is injected into the page ( into the #content div )


Modules

In every view module you can have an init( ) and a destroy( ) that you can use to set up / clean up your scripts respectively. If you use the "EventMaster.js" module included, you don't need to remove any event listeners that you didn't explicitly set to permanent.

You can have none, 5 or 1 million bazillion of these if you want, they will all be handled behind the scenes. Because there is no way to unload a script from memory, it would be wise to go easy, although they will only run on the page they live in.


* Make sure to export the functions here - ie:

export function init(){
  VARS['awesome'] = true;
}


export function destroy(){
  console.log(`I am ${ VARS.awesome ? 'still' : 'not' } awesome, and about to be destroyed.`);
}


You can use global variables here if you want, but there is a recommended way of setting variables so the memory gets released by GC every page:

Use the global object "VARS" to store data. This is a global object that gets overwritten every page load.


Example:

VARS['variable_name'] = 5

Then you can access it later:

console.log( VARS['variable_name'] );
/** OR **/
console.log( VARS.variable_name );
// etc
VARS.variable_name += 48;

I like this way personally ( besides having to use strings ) because you can quickly see where we are declaring variables vs. where we are accessing them. Like I said already, you can store almost anything you want in VARS ( Functions, variables, other Objects, etc. anything that can be a property of an Object )


CSS Files

These are your standard, VANILLA .css files. The only thing I would note is to be careful what you name your classes/Ids as there will be situations where any .css file will live with another .css file, and you could run into problems.


I would reccomend you just use a prefix here. ex:

.name-of-file-title{
}



Custom Events

There are a few custom events you can listen for ( on document.body ):


'page_unload' - Fires when the old page is unloaded ( VARS.loaded_page_anchor is the old anchor )

'page_load' - Fires when the new page is loaded ( VARS.loaded_page_anchor is the new anchor )

'page_animated' - Fires when the page is done animating
